The test suite org.objectweb.carol.irmi.test.RMISuite starts two
external processes: the RMI registry and the server process that
exports remote objects.  To make debugging easier, this commit makes

r568 | el-vadimo | 2005-09-27 10:18:40 -0600 (Tue, 27 Sep 2005) | 30 lines

Reverted the change made in
src/org/objectweb/carol/irmi/RMIObjectOutputStream.java,v1.3

Put in a fix for the failing test
ReferentialIntegrityTest#testRemoteParameterIntegrity

The issue is that RMIObjectOutputStream#replaceObject may be called on
the _client_side_ and passed a RemoteStub.  This happens when the
client is marshalling parameters of a remote method call, some of
which may happen to be remote objects.  In this case, iterating
through Server#getServers() and calling Server#getStub(Remote) is
pointless, because there are no servers.  Since we are holding a
RemoteStub instance in our hands already, we should just return it as
is.

This commit may leave room for further micro-optimizations.  There are
three possibilities

  1. obj is an instance of RemoteStub
  2. obj is not an instance of RemoteStub, but is an instance of Remote
  3. obj is not an instance of Remote.

To micro-optimize, we would want to check for the most likely case
first.  I'm not sure what the most frequence case actually is.

Anyhow, with this change in place, all the tests are passing

    Running org.objectweb.carol.irmi.test.RMISuite
    Tests run: 19, Failures: 0, Errors: 0, Time elapsed: 2.965 sec

r560 | el-vadimo | 2005-09-22 11:46:43 -0600 (Thu, 22 Sep 2005) | 52 lines

See http://www.objectweb.org/wws/arc/carol/2005-09/msg00003.html

Removed the interface
org.objectweb.carol.rmi.jrmp.server.JUnicastThreadFactory

The only two places where this interface was referenced were:
  src/org/objectweb/carol/rmi/jrmp/server/JUnicastRemoteObject.java
  src/org/objectweb/carol/rmi/jrmp/server/JUnicastServerRef.java

These two files have been cleaned up in this commit.  The inner class
JUnicastServerRef.DispatchRunnable was also removed.

Technically, this change breaks the API, because the following two
public methods are removed from the public class
org.objectweb.carol.rmi.jrmp.server.JUnicastRemoteObject:

    static void setDefaultThreadFactory(JUnicastThreadFactory factory);
    static JUnicastThreadFactory getDefaultThreadFactory();

In practice, this shouldn't matter because the only known current user
of Carol is Jonas, and Jonas doesn't reference the affected Carol
classes anywhere in its code.

Moreover, there couldn't have been any other users of this removed
code, because it was broken anyway.  Look at the diff between
JUnicastServerRef.java v1.7 and v1.8:


- JUnicastThreadFactory factory = JUnicastRemoteObject.getDefaultThreadFactory();
- if (factory == null) {
-     runDispatch(obj, call);
- } else {
-     DispatchRunnable dr = new DispatchRunnable(obj, call);
-     factory.getThread(dr).run(); // run the target
-     if (dr.getIOException() != null) throw dr.getIOException();
- }

Currently, 'factory==null' is always true, because the
setDefaultThreadFactory method is never called -- neither in Carol,
nor in Jonas.  Assume for the sake of argument that there was someone
out there who used Carol outside of Jonas, setting the 'factory'
variable to a non-null value.  Then they would end up running the
'else' branch of the above conditional which executes this code:

   factory.getThread(dr).run();

However, this code is bogus, because you are not supposed to ever call
the Thread#run() method directly.  You have to call Thread#start().
So, this branch of conditional is currently broken (and always has
been).  Since no one has complained about it to date, no one uses
it. Q.E.D.

r519 | el-vadimo | 2005-04-28 12:21:51 -0600 (Thu, 28 Apr 2005) | 33 lines

Made the getJRMPInitializers() method thread-safe.

The method wasn't thread-safe for the following reasons.

It lazily instantiates the static "initializers" array. The pared down
outline of this method looks like so:

    public static String[] getJRMPInitializers() {
        // if exists, return it
        if (initializers != null) {
            return initializers;
        }
        // else, compute it.
        // ...
        initializers = new String[sz];
        for (int i = 0; i < sz; i++) {
            initializers[i] = (String) ins.get(i);
        }
        return initializers;
    }

Suppose Thread A comes along and finds the "initializers" field to be
null.  So, Thread A executes

       initializers = new String[sz];

and begins populating the newly allocated array.

Suppose further that while Thread A is still populating the
"initializers" array, Thread B comes along.  It finds "initializers"
to be non-null.  So, in Thread B, this method returns a partially
initialized value of this field.  Not good.

r518 | el-vadimo | 2005-04-28 12:17:54 -0600 (Thu, 28 Apr 2005) | 36 lines

Removed the useless private static field "init".

Since the "init" field is private, we know it wasn't used anywhere
outside the JInterceptorStore class.  Inside the class, it was
referenced three times.  The following pared down skeletal version of
the class shows all of the three occurrences:

 | public class JInterceptorStore {
 |
 |     private static boolean init = false;  // 1st mention
 |
 |     static {
 |         if (!init) {                      // 2nd mention
 |             try {
 |                 // do stuff that may throw an exception
 |                 init = true;              // 3rd mention
 |             } catch (Exception e) {
 |                 TraceCarol.error("JrmpPRODelegate(), No interceptors found", e);
 |             }
 |         }
 |
 |     }
 |
 |     // init is never referenced again below
 | }

The "if (!init)" test was obviously redundant, since we know that
"init" was false at that point, therefore "!init" was true, and the
condition's body was _always_ executed.

Resetting the "init" to true inside the try-catch statement was
redundant, because the value of "init" was never referenced after
that.

Because of all that, the field was safe to remove.

r481 | benoitf | 2005-04-07 09:07:09 -0600 (Thu, 07 Apr 2005) | 33 lines

Change the configuration process of Carol as discussed here:
http://www.objectweb.org/wws/arc/carol/2005-03/msg00047.html
http://www.objectweb.org/wws/arc/carol/2005-03/msg00050.html
etc.

The configuration model is the following :
Instead of RmiConfiguration CarolConfiguration and CarolCurrentConfiguration.

There is only one entry named : ConfigurationRepository.

On this class, we can set a current configuration (ProtocolConfiguration)
ConfigurationRepository is also a factory of ProtocolConfiguration object (default imp =  ProtocolConfigurationImpl).
ConfigurationRepository.newConfiguration(protocolName, configName)

Protocol class is used by ProtocolConfiguration classes.

ie : jrmp is a Protocol class with its nameservice, PROdelegate, etc
the configuration "jrmp" use jrmp protocol but specify the PROVIDER_URL for example
So we can have jrmp1 with a PROVIDER_URL dummy:2001 and jrmp2 with PROVIDER_URL:dummy:2002


To retrieve the current protocol :
 ConfigurationRepository.getCurrentConfiguration().getProtocol().getName();

We can also initialize carol with an URL file : (#303299 feature reques)
 ConfigurationRepository.init(URL object).

Also we can use :
Properties p = protocolConfiguration.getProperties();
p.put(key,value)
protocolConfiguration.configure(p);
for feature request : #303300

Refactoring of the spi package.

Before the changes :
 - Each class handle its own cache. it should be the factory role
 - MultiOrbInitialContextFactory redirect to ContextWrapper and then on MultiContext and then on the wrapped context. (We could remove one wrapper layer)
- A lot of duplicate code for factories and Context implementation
- The jrmp local context is a wrapper/ handle its cache and wrapped on a registry object / would be better to split the roles of this class to make it simpler


After the changes :
 - names of factory have been kept (useful for backward compliance)
 - Add of an abstract context and an abstract factory doing a lot of commons code
- Context wrapper extends the abstract context and have to implement the wrapping of special objects like Serializable objects, etc.
- Factories extends the abstract factory and have to implement some abstract methods
- The cache mechanism is now only in the abstract factory instead of having it one by Context class. The factories build contexts and handle the cache. This simplify the role of the wrapper, the maintenance, the bugs...
- Exceptions should not be lost anymore. There is still a problem on NoSuchObjectException as we cannot set the initcause on RemoteException.
- Remove ContextWrapper class. The encoding is now done in abstract context.
So we avoid a useless wrapper. This is simpler.
- A lot of checkstyle/javadoc to be compliant with .checkstyle code convention.

This fixes two occurrences of the following problematic code pattern
in CAROL.

  try {
      // perform some actions
  } catch (Exception e) {
      String msg = "some useless message: " + e;
      throw new NamingException(msg);
  }

This is bad for the following reasons:

 * It's a bad idea to catch unchecked exceptions, except in those
   cases where code is expected to throw a specific subclass of
   RuntimeException (like Integer.parseInt(String) which is documented
   to throw the unchecked exception NumberFormatException).  Only
   checked exceptions should be caught.

 * Once an exception is caught, care must be taken to preserve the
   original stack trace.  In the above example, the correct thing to
   do is

     try {
         // perform some actions
     } catch (SomeCheckedException e) {
         throw (NamingException) new NamingException("some message").initCause(e);
     }

   This uses the Throwable#initCause(Throwable) method to properly
   chain exceptions.

  * In this particular case, the only checked exception that is thrown
    in the "try {}" block is NamingException.  There is little value
    in wrapping a NamingException inside another naming exception.
    Instead, the original exception should be allowed to propagate.
    In other words, the try-catch block can be eliminated entirely in
    many places in CAROL where it is currently used.

In this particular case, the original exception is discarded twice.

The original log message is reproduced here for convenience:

..............................................................................
This change affects the following two files:

  test/conform/generic/carol_double.xml
  test/src/org/objectweb/carol/jtests/conform/util/ProcessRunner.java

It fixes intermittent test failures described in this commit message:

http://cvs.forge.objectweb.org/cgi-bin/viewcvs.cgi/carol/carol/test/conform/generic/carol_double.xml?rev=1.11.2.2&content-type=text/vnd.viewcvs-markup

For more background info, see also
http://mail-archive.objectweb.org/carol/2005-01/msg00000.html

These intermittent failures were occurring due to the following
reasons.

There are currently two kinds of tests in CAROL: "single" and
"multi".  In "single" tests, three procecess are started: a
registry, the server, and the client.  In "multi" tests, five
processes are started concurrently: two registries, one remote server,
and two remote clients.

Only "multi" tests were experiencing intermittent failures.  These
failures were occurring due to the following asymmetry between the two
clients.

In order to successfully execute "multi" tests, the aforementioned
five processes have to be started and stopped in the correct order.
The startup order is as follows:

 (1) the two registries must be started;

 (2) once the registies are up and running, the remote server must be
     started;

 (3) once the server is running, the two clients must be started.

In our case, the clients are JUnit processes that execute and test
remote method calls.  The clients terminate on their own without
requiring any special external intervention.  In contrast, the server
and the registry must be explicitly shut down by the test harness.

The shutdown must occur in the reverse order.

 (1) once _both_ clients are finished running, the server must be
     shutdown.

 (2) once the server is down, the registries must be shut down.

The bug causing intermittent failures was present in carol_double.xml.
The problem was that we were shutting down the server as soon as the
first client was finished.  Depending on whether or not the second
concurrently executing client was still running at the time of remote
server shutdown, intermittent failures might or might not occur.

The fix is to make sure that the remote server is only shut down after
_both_ clients have exited.

For convenience, the original log message is reproduced below:

..............................................................................

The method isStarted() checks whether a name server is already
running.  The check is performed by attempting to instantiate a new
InitialContext.  This instantiation attempt will raise an exception,
if the name server is not running.  The previous implementation of
this idea had the following flaw.

Under Sun's JVM, attempting to instantiate a new InitialContext in the
absence of a running name server results in a
javax.naming.CommunicationException.  The previous implementation of
isStarted() would catch this exception and correctly return "false".

This doesn't work under IBM's JDK.  When no name server is running,
instantiating a new InitialContext results in a
javax.naming.ConfigurationException under IBM's JDK.  Under the
previous implementation of isStarted(), this case would follow a
different execution path.  The ConfigurationException would be caught
under the catch-all clause "catch (Exception e)" and the method would
return "true", _erroneously_ indicating that the name server *is*
already running.

This patch introduces two changes:

 (a) it removes the highly questionable catch-all clause
     "catch (Exception e)"

 (b) instead of catching javax.naming.CommunicationException, we now
     catch javax.naming.NamingException.

     This is correct because both CommunicationException thrown by Sun
     and ConfigurationException thrown by IBM are subclasses of
     NamingException.  As a matter of fact, the constructor
     InitialContext(Properties) is documented to throw
     NamingException.  Catching a specific subclass of NamingException
     is highly non-portable.

The purpose of this commit is to change the way we set up and tear
down multiple concurrent processes when running unit tests.

For background information, see these threads:

 - http://mail-archive.objectweb.org/carol/2004-12/msg00005.html
 - http://mail-archive.objectweb.org/carol/2004-12/msg00022.html

The unit tests verify correctness of various RMI implementations
provided by CAROL.  To test remote method calls, we normally need
three different processes running concurrently:

  (a) registry;
  (b) remote server;
  (c) remote client.

(In actuality, this ranges from 2 to 5 concurrent processes under
different scenarios that CAROL currently tests for.)

These processes should be started in the above order and shut down in
the reverse order.  Prior to this commit, CAROL had no _precise_ way
of starting these processes in the right order and tearing them down
correctly.  The desired effect was achieved by using Ant's <parallel>
task in concert with <sleep> to start the processes in the right
order.  For teardown, we were relying on the 'timeout' attribute of
the <java> task.  One of the side effects of this approach was that
the tests took 9 minutes to run regardless of how fast your computer
was.

This commit eliminates the use of sleeps and timeouts.  Instead, we
now rely on the Ant's <waitfor> task to ensure that the required
processes are started in the right order.  Two new helper classes are
introduced to allow us to programmatically shut down the processes by
sending a command over TCP.  These two new classes are
  org.objectweb.carol.jtests.conform.util.ProcessRunner
  org.objectweb.carol.jtests.conform.util.ProcessStopper

The above accounts for the bulk of the changes committed in this
checkin.

Another small change was to _not_ generate javadocs as part of running
tests.

The end result is that tests now take anywhere from 1 to 3 minutes to
finish depending on how fast your machine is.  This is down from 9
minutes.

This commit introduces a seeming regression.  Tests involving either
jrmp1.1 or jrmp1.2 now fail intermittently.  For tests that do not
involve either of these two protocols, there are no regressions.  I
will investigate intermittent failures and commit a fix as soon as I
have it.
